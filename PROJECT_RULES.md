# Pin 项目文档阅读规则

> 本规则为 Pin 拼豆应用项目的执行规范，所有 AI 辅助开发必须遵循此规则。

## 项目概述

Pin 是一款拼豆创作应用，用户可以在网格画板上选择颜色、放置拼豆进行创作。基于 uni-app 跨平台框架（Vue3 + TypeScript）构建，支持 Web、Android、iOS、小程序多端部署，数据存储采用本地存储 + 云端同步方案。

## 技术栈

- **框架**：uni-app（Vue3 + Composition API + TypeScript）
- **状态管理**：Vue3 ref/reactive（后续可引入 Pinia）
- **UI 组件**：uni-ui / 自定义组件
- **构建工具**：Vite（uni-app 内置）
- **多端支持**：H5、App（iOS/Android）、微信小程序、支付宝小程序

## 阅读规则

### 规则 1：版本优先级
- 始终优先阅读最新版本的文档
- 历史版本文档仅在做版本对比或回溯问题时参考
- 需求变更以最新版本需求文档为准，历史版本中已废弃的需求不再适用

### 规则 2：文档阅读顺序
开发新功能或修复 BUG 时，按以下顺序阅读：
1. **目录索引** → 了解整体文档结构
2. **最新需求文档**（01-需求文档）→ 理解功能需求
3. **UI/UE 设计规范**（02-设计文档）→ 确认视觉和交互标准
4. **数据文档**（03-技术文档）→ 了解数据模型和存储结构
5. **多端适配文档**（03-技术文档）→ 了解各端差异和适配方案
6. **最新测试用例**（04-测试文档）→ 了解已有测试覆盖

### 规则 3：文档命名规范
所有文档标题统一格式：`Pin - [版本号] [文档类型]`
- 版本号：MVP V1.0 / V1.1 / V1.2.0 等
- 文档类型：需求文档 / 测试用例 / 测试报告 / 设计规范 / 数据文档 / 多端适配文档
- 新增文档时必须遵循此命名规范

### 规则 4：完整研发流程（强制执行）
任何新功能或重大改动必须严格按以下阶段执行，每个阶段有明确的输入、输出和确认节点：

```
阶段 1：需求对齐
  输入：用户原始需求 / BUG 反馈
  输出：完整需求文档（含功能描述、交互说明、边界条件）
  确认：产品负责人确认需求文档 ✅

阶段 2：UI/UE 界面确认
  输入：已确认的需求文档
  输出：UI/UE 设计稿或界面方案确认
  确认：UI 和 UE 同学确认界面方案 ✅

阶段 3：开发 + 开发自测
  输入：已确认的需求文档 + UI/UE 方案
  输出：可运行的代码实现
  确认：开发自测通过 ✅
  说明：开发完成后需进行基础自测，确保页面可访问、核心功能可用

阶段 4：AI 编写测试用例 + AI 执行测试
  步骤 4.1：AI 编写测试用例
    输入：已确认的需求文档 + 已完成的代码实现
    输出：测试用例文档（输出至飞书 04-测试文档文件夹）
    内容：功能测试用例、边界条件用例、UI 交互用例，按优先级分为 P0/P1/P2
  步骤 4.2：AI 执行测试
    输入：测试用例文档 + 可运行的构建版本
    输出：测试报告（输出至飞书 04-测试文档文件夹）
    内容：各端通过/失败记录、BUG 列表（含复现步骤）、截图证据
  确认：所有 P0/P1 用例在目标平台通过 ✅
    - H5 端测试（MVP 阶段优先）
    - 微信小程序端测试（后续版本）
    - Android 端测试（后续版本）
    - iOS 端测试（后续版本）

阶段 5：产品/UI&UE 验收
  输入：测试通过的构建版本
  输出：验收结论（通过 / 打回）
  确认：产品 + UI + UE 均确认通过 ✅

阶段 6：发布
  输入：验收通过的代码
  输出：各端正式环境部署完成
  确认：
    - H5：CI 构建成功 + 线上可访问 ✅
    - 小程序：审核通过 + 发布上线 ✅
    - App：应用商店提审/上架 ✅

阶段 7：线上测试回归
  输入：已发布的正式环境
  输出：线上回归测试结论
  确认：核心流程在目标平台验证通过 ✅
```

**关键原则：**
- 每个阶段必须收到确认后才能进入下一阶段
- 阶段 1 和阶段 2 是前置条件，未经确认不得开始开发
- 阶段 5 验收未通过时，打回至阶段 3 重新开发
- 阶段 7 发现问题时，按 BUG 记录规范（规则 6）记录并进入下一轮修复流程
- 紧急修复（如线上 P0 BUG）可简化流程，但至少保留：开发 → 测试 → 发布 → 线上回归

#### 7.1 GitHub Pages 部署问题解决方案（重要）

**问题现象**：页面显示 "连接服务器超时，点击屏幕重试"

**根本原因**：uni-app H5 默认对非 tabBar 页面使用懒加载，浏览器缓存了旧的 index.html，引用了不存在的 JS chunk 文件，导致 404 错误。

**解决方案**：在 `vite.config.js` 中禁用懒加载，将所有代码打包到单个 JS 文件：

```javascript
import { defineConfig } from 'vite'
import uni from '@dcloudio/vite-plugin-uni'

export default defineConfig({
  base: '/',
  plugins: [uni()],
  build: {
    rollupOptions: {
      output: {
        // 禁用动态导入，将所有代码打包到一个文件
        inlineDynamicImports: true,
        entryFileNames: 'assets/[name]-[hash].js',
      },
    },
  },
})
```

**验证方法**：构建后检查 `dist/build/h5/assets/` 目录，应该只有 1 个 JS 文件（`index-xxx.js`），而不是多个 `pages-xxx.js` 文件。

**故障排查**：
1. 检查 GitHub Pages 部署状态：`curl -s "https://api.github.com/repos/CrazyDave0514/Pin/pages"`
2. 检查文件是否存在：`curl -I "https://pindou.picsync.cn/assets/index-xxx.js"`
3. 浏览器控制台查看是否有 `ERR_ABORTED` 或 `Failed to fetch` 错误

### 规则 9：文档存放位置规范（强制执行）
**所有项目文档必须存放在飞书云空间，严禁仅保存在本地。**

#### 9.1 强制要求
- ✅ **唯一数据源**：飞书云空间为所有项目文档的唯一正式存储位置
- ❌ **禁止本地-only**：任何文档不得仅保存在本地而不上传飞书
- ✅ **本地仅作缓存**：本地文件仅作为临时编辑缓存，完成后必须上传飞书

#### 9.2 文档创建流程
```
方式 1：直接在飞书创建（推荐）
  → 在飞书云空间对应文件夹内新建文档
  → 编辑完成后自动保存

方式 2：本地编辑后上传
  → 在本地用 Markdown 编辑
  → 完成后通过 lark-cli 导入飞书
  → 确认飞书版本正常后，本地文件可删除或作为备份
```

#### 9.3 文档更新流程
1. **读取**：从飞书读取当前文档内容
2. **编辑**：在本地或飞书在线编辑
3. **保存**：更新后的内容必须保存回飞书
4. **确认**：确认飞书版本已更新，与本地一致

#### 9.4 违规处理
- 发现文档仅存在于本地而未上传飞书 → **必须立即上传**
- 飞书版本与本地版本不一致 → **以飞书版本为准**，本地版本作为参考
- 多人协作时以飞书版本为唯一可信源

#### 9.5 例外情况
以下文件可仅保存在代码仓库：
- 技术配置文件（如 `.eslintrc`, `tsconfig.json`）
- README.md（项目根目录说明）
- 代码注释和文档字符串

**所有产品、设计、需求、测试等业务文档必须在飞书管理。**

### 规则 10：多端开发规范
- **代码复用优先**：核心业务逻辑、数据层、工具函数必须跨端复用
- **平台差异隔离**：各端差异化代码使用条件编译处理
- **条件编译**：使用 uni-app 的条件编译语法处理平台差异
  ```typescript
  // 示例
  // #ifdef H5
  // H5 特有逻辑
  // #endif
  // #ifdef MP-WEIXIN
  // 微信小程序特有逻辑
  // #endif
  ```
- **样式适配**：
  - 使用 `rpx` 作为响应式单位
  - 安全区域使用 `safe-area-inset-*` CSS 变量
- **API 适配**：使用 uni-app 统一 API（`uni.xxx`），避免直接调用平台原生 API
- **数据存储**：使用 `uni.getStorageSync` / `uni.setStorageSync`，避免直接使用 `localStorage`
